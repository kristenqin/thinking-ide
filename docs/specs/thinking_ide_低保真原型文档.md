# Thinking IDE P0 低保真原型

## 1. 文档信息

| 字段 | 内容 |
|---|---|
| 产品名称 | Thinking IDE |
| 文档类型 | P0 低保真原型 / Low-fidelity Wireframe |
| 版本 | v0.2 |
| 阶段 | MVP |
| 原型范围 | 5 张 P0 核心状态 |
| 技术约束 | React + React Flow + shadcn/ui / Radix UI |

## 2. 原型目标

本低保真原型用于验证 Thinking IDE 的核心交互闭环：

```text
官方 Chat 页面右侧注入 Thinking Panel
↓
用户正常聊天
↓
AI 回复完成后生成 Concept Map 草稿
↓
用户直接拖拽、重命名、连接、删除节点
↓
用户点击节点跳转原文
↓
Concept Map 与聊天思考过程同步推进
```

本阶段不关注最终视觉风格，只关注：

1. 页面区域是否合理。
2. 右侧面板是否能承载 Concept Map。
3. 节点是否足够轻量，能突出概念本身。
4. 用户是否能通过直接操作整理节点。
5. 原文跳转是否能形成清晰反馈。

## 2.1 当前 MVP 原型策略补充

当前 MVP 原型优先验证：

1. `Session -> Question -> Answer -> Outline` 的结构树主视图是否比 graph 更容易浏览长会话。
2. 用户是否能自然地展开/收起问题、回答和大纲层级。
3. `Concept View` 保留为后续增强视图，不作为当前低保真主原型的第一优先对象。

## 3. P0 原型清单

| 编号 | 原型名称 | 验证重点 |
|---|---|---|
| WF-01 | 官方 Chat + Thinking Panel 默认展开 | 4:6 空间关系、面板整体结构 |
| WF-02 | Thinking Panel 空状态 | 首次打开时的用户引导 |
| WF-03 | 结构树草稿生成成功状态 | 问题/回答/大纲层级、展开/收起、底部弱统计 |
| WF-04 | 节点 selected + Floating Toolbar 状态 | 用户如何直接操作节点 |
| WF-05 | 节点跳转原文高亮状态 | 右侧节点与左侧 Chat 原文联动 |

# WF-01 官方 Chat + Thinking Panel 默认展开

## 1. 场景说明

用户打开官方 AI Chat 页面后，Thinking IDE 自动在右侧注入 Thinking Panel。左侧 Chat Area 承载原始对话，右侧 Thinking Panel 作为主要认知建模空间。

Thinking Panel 不是窄辅助栏，而是 Concept Map 工作台，因此默认采用 4:6 布局：

```text
Official Chat Area：40%
Thinking Panel：60%
```

## 2. 低保真线框

```text
┌──────────────────────────────┬──────────────────────────────────────────────┐
│                              │ Thinking IDE        Synced      ↻   ⚙   ⟩     │
│      Official Chat Area      ├──────────────────────────────────────────────┤
│                              │                                              │
│ ┌──────────────────────────┐ │                                              │
│ │ User                     │ │              Concept Map Canvas              │
│ │ 我想设计一个用于构建认知   │ │                                              │
│ │ 模型的工具……             │ │       ┌──────────────┐                       │
│ └──────────────────────────┘ │       │ 认知模型工具  │                       │
│                              │       └──────┬───────┘                       │
│ ┌──────────────────────────┐ │              │ 回应                          │
│ │ Assistant                │ │              ▼                               │
│ │ 这个工具可以理解为一个 AI │ │       ┌──────────────┐                       │
│ │ Chat 右侧的 Concept Map… │ │       │ Concept Map  │                       │
│ │                          │ │       │ 工作台        │                       │
│ │ 1. 它不是重做 Chat       │ │       └──────────────┘                       │
│ │ 2. 它辅助构建概念地图     │ │                                              │
│ │ 3. 用户拥有最终控制权     │ │                                              │
│ └──────────────────────────┘ │                                              │
│                              ├──────────────────────────────────────────────┤
│ ┌──────────────────────────┐ │ Restored current map                         │
│ │ Message input...         │ │                                              │
│ └──────────────────────────┘ │                                              │
└──────────────────────────────┴──────────────────────────────────────────────┘
```

## 3. 结构说明

### 左侧 Official Chat Area

保留官方 Chat 原有能力：

1. 用户消息展示。
2. AI 回答展示。
3. 官方输入框。
4. 官方滚动行为。

Thinking IDE 不重做左侧 Chat。

### 右侧 Thinking Panel

右侧默认展开，包含：

1. Header 顶部栏。
2. Concept Map Canvas。
3. React Flow 节点图区域。
4. Bottom Log 底部弱提示。

## 4. Header 元素

```text
Thinking IDE        Synced      ↻   ⚙   ⟩
```

| 元素 | 含义 |
|---|---|
| Thinking IDE | 面板标题 |
| Synced | 当前状态，表示已同步 |
| ↻ | 重新解析当前轮回答或当前选中来源 |
| ⚙ | 设置入口，包含语言切换 |
| ⟩ | 收起右侧面板 |

设置入口包含：

```text
Settings
├── Language
│   ├── 中文
│   └── English
├── Auto-generate
├── Clear current map
└── About
```

## 5. 默认交互

1. 点击 `⟩`：右侧面板收起为窄条。
2. 点击 `↻`：对当前选中 answer node 或最近一轮回答重新解析。
3. 点击节点：选中节点，并尝试跳转到左侧对应原文。
4. 双击节点标题：重命名。
5. 拖拽节点：调整 Concept Map 布局。
6. 从节点连接点拖线：创建关系。
7. 点击关系线：打开关系编辑浮层。
8. Delete / Backspace：删除选中节点或关系。

## 6. 验证点

1. Thinking Panel 60% 宽度是否能提供足够思考空间。
2. 左侧 Chat 40% 是否仍能正常阅读和输入。
3. Header 是否保持简洁。
4. 用户是否能理解右侧是 Concept Map 工作区。
5. 底部弱提示是否不干扰主思考区域。

# WF-02 Thinking Panel 空状态

## 1. 场景说明

用户打开一个新 Chat 页面，当前还没有可解析的问答内容。Thinking Panel 显示空状态，引导用户开始对话。

```text
│                              │   你可以：                                    │
│                              │   · 查看 AI 生成的节点草稿                     │
│                              │   · 点击节点回到原文                           │
│                              │   · 拖拽、重命名、连接和删除节点                 │
│                              │                                              │
│ ┌──────────────────────────┐ │                                              │
│ │ Message input...         │ │                                              │
│ └──────────────────────────┘ │                                              │
└──────────────────────────────┴──────────────────────────────────────────────┘
```

## 3. 空状态文案

主文案：

```text
开始一段对话后，Thinking IDE 会在这里生成 Concept Map 草稿。
```

辅助文案：

```text
你可以：
· 查看 AI 生成的节点草稿
· 点击节点回到原文
· 拖拽、重命名、连接和删除节点
```

## 4. 状态规则

触发空状态的条件：

1. 当前页面没有可解析问答。
2. 当前 conversationKey 没有本地 Concept Map 数据。
3. Chat Adapter 注入成功，但暂未监听到消息。

## 5. 用户下一步

用户可以：

1. 在左侧 Chat 输入问题。
2. 收起 Thinking Panel。
3. 等待 AI 回复完成后自动生成 Concept Map 草稿。

## 6. 验证点

1. 用户是否知道下一步应该开始聊天。
2. 用户是否能理解右侧后续会生成 Concept Map 草稿。
3. 空状态是否不会打扰左侧 Chat 的使用。

# WF-03 Concept Map 草稿生成成功状态

## 1. 场景说明

用户完成一轮问答后，AI 回复已经结束。Thinking IDE 自动生成 question node、answer node、answer_outline node、concept node 和关系。

该状态需要验证：节点是否足够轻量，画布是否把注意力放在概念和关系本身，而不是系统属性上。

```text
┌──────────────────────────────┬──────────────────────────────────────────────┐
```

## 3. 画面说明

### 顶部 Header

顶部只显示产品名、当前状态和工具入口。

不显示：

```text
本轮新增 8 个候选节点，4 条候选关系
```

原因：节点和关系数量是系统执行统计，不应该占据主要注意力。

### 底部弱提示

底部展示轻量生成日志：

```text
Generated 8 nodes · 4 relations
```

这类信息仅作为系统反馈，不参与主视觉。

### 节点视觉

节点默认只显示概念标题：

```text
┌──────────────┐
│ Concept Map  │
└──────────────┘
```

不默认展示：

1. Role 标签。
2. Candidate / Edited 状态。
3. 原文按钮。
4. 接受 / 忽略按钮。
5. 多余属性信息。

### AI 新生成节点的弱提示

如果需要区分 AI 新生成节点，可以用弱虚线或小圆点：

```text
┌ - - - - - - - ┐
│ AI草稿      · │
└ - - - - - - - ┘
```

## 4. 用户可执行操作

1. 点击节点：选中并尝试定位原文。
2. 双击标题：重命名。
3. 拖拽节点：调整位置。
4. 从节点连接点拖线：创建关系。
5. 点击关系线：编辑关系。
6. Delete / Backspace：删除选中节点或关系。
7. 选中节点后显示 Floating Toolbar。

## 5. 验证点

1. 用户注意力是否集中在概念和关系本身。
2. 节点是否足够轻量。
3. 不显示 role / status 是否更接近 Concept Map 工具。
4. 底部统计是否足够弱化。
5. 用户是否能理解节点可以直接操作。

用户点击某个节点后，节点进入 selected 状态，并在节点附近显示 Floating Toolbar。高频操作通过工具条直接完成，低频操作进入更多菜单。

## 3. Floating Toolbar 操作

```text
重命名  原文  连接  ⋯
```

| 操作 | 结果 |
|---|---|
| 重命名 | 节点进入标题编辑态 |
| 原文 | 跳转到 Chat 原文 |
| 连接 | 开启节点连接模式 |
| ⋯ | 打开更多操作菜单 |

## 4. 直接操作规则

除 Floating Toolbar 外，节点还支持直接操作：

| 操作 | 结果 |
|---|---|
| 点击节点 | 选中节点，并尝试定位原文 |
| 双击标题 | 重命名 |
| 拖拽节点 | 移动节点 |
| 从连接点拖线 | 创建关系 |
| Delete / Backspace | 删除选中节点 |

## 5. More Menu 低保真

点击 `⋯` 后：

```text
┌─────────────────────┐
│ 转为 Concept          │
│ 转为 Claim            │
│ 重新解析来源           │
│ 查看属性              │
│ 删除节点              │
└─────────────────────┘
```

## 6. 设计原则

1. Dropdown 不是主操作入口。
2. 高频操作通过直接操作和 Floating Toolbar 完成。
3. 更多菜单只承载低频操作。
4. 节点不显示“接受 / 忽略”按钮。
5. 用户不需要逐个确认 AI 生成节点。

## 7. 验证点

1. 用户是否能自然理解选中节点后的操作方式。
2. Floating Toolbar 是否比 Dropdown 更有自由操作感。
3. 节点主视觉是否仍然保持轻量。
4. 操作入口是否不遮挡过多画布空间。

# WF-05 节点跳转原文高亮状态

## 1. 场景说明

用户点击右侧节点，左侧 Chat Area 自动滚动到对应用户提问、AI 回答或回答段落，并短暂高亮。右侧节点保持 selected 状态。

这是 Thinking IDE 解决长对话定位成本的核心交互。

## 2. 低保真线框

```text
┌──────────────────────────────┬──────────────────────────────────────────────┐
│                              │ Thinking IDE        Synced      ↻   ⚙   ⟩     │
│      Official Chat Area      ├──────────────────────────────────────────────┤
│                              │ 已定位到原文                                  │
│ ┌──────────────────────────┐ ├──────────────────────────────────────────────┤
│ │ Assistant                │ │                                              │
│ │                          │ │              Concept Map Canvas              │
│ │ 这个方向可以定义为 AI 辅助 │ │                                              │
│ │ 用户主导的 Concept Map…  │ │       ┌──────────────┐                       │
│ │                          │ │       │ 产品定位       │                       │
│ │ 1. 用户主导              │ │       └──────┬───────┘                       │
│ │                          │ │              │ 回应                          │
│ │ ┌──────────────────────┐ │ │              ▼                               │
│ │ │ 2. AI 生成草稿        │ │ │       ╔══════════════╗                       │
│ │ │ AI 负责从聊天中提取   │ │ │       ║ Concept Map  ║                       │
│ │ │ 节点和关系，用户负责   │ │ │       ╚══════════════╝                       │
│ │ │ 重命名、连接和删除。   │ │ │          │    │ 包含                          │
│ │ └──────────────────────┘ │ │          ▼    ▼                               │
│ │                          │ │      ┌──────┐ ┌──────┐                       │
│ │ 3. 节点和关系可编辑       │ │      │用户主导│ │AI草稿 │                       │
│ └──────────────────────────┘ │      └──────┘ └──────┘                       │
│                              │                                              │
│                              ├──────────────────────────────────────────────┤
│                              │ Located source                               │
└──────────────────────────────┴──────────────────────────────────────────────┘
```

## 3. 状态说明

### 左侧 Chat Area

被定位的原文段落：

```text
短暂高亮 1.5 到 3 秒
```

高亮对象：

1. question node：用户消息卡片。
2. answer node：AI 回答整体。
3. answer_outline node：AI 回答中的目标段落。
4. concept node：对应来源段落或来源消息。

### 右侧 Thinking Panel

对应节点进入 selected 状态：

```text
加粗边框 / selected 标识
```

状态条显示：

```text
已定位到原文
```

## 4. 跳转交互规则

```text
用户点击节点
↓
读取 source.messageRefId / anchor
↓
Chat Adapter 定位 DOM
↓
滚动到目标区域
↓
目标区域高亮
↓
右侧节点 selected
```

## 5. 定位失败兜底

如果 answer_outline 的精确 anchor 找不到：

```text
滚动到 AI 完整回答
↓
提示：精确段落无法定位，已跳转到对应回答
```

如果 messageRef 完全找不到：

```text
节点显示 ⚠
↓
Toast：原始消息位置暂时无法定位，节点内容仍已保留。
```

## 6. 验证点

1. 用户是否能感知“点击节点 → 回到原文”。
2. 原文高亮是否足够明显。
3. 右侧 selected 状态是否能帮助用户建立左右对应关系。
4. 这个交互是否确实降低长对话定位成本。

# 6. P0 原型闭环总结

这 5 张 P0 低保真原型覆盖 Thinking IDE 的最小体验闭环：

```text
WF-01 默认展开
↓
WF-02 空状态
↓
WF-03 生成 Concept Map 草稿
↓
WF-04 直接操作节点
↓
WF-05 点击节点跳转原文
```

它们共同验证：

1. Thinking Panel 能否自然嵌入官方 Chat 页面。
2. 4:6 布局是否能提供足够的思考空间。
3. 用户是否能理解右侧是 Concept Map 工作区。
4. AI 自动生成的节点是否足够轻量。
5. 用户是否能通过直接操作整理节点。
6. 用户是否能通过右侧节点快速回到左侧原文。
7. Concept Map 构建是否能与聊天思考过程同步发生。

确认这 5 张 P0 原型后，再继续补充 P1 原型：

1. 节点标题编辑状态。
2. 关系编辑状态。
3. concept node 多来源选择状态。
4. 生成失败 / DOM 识别失败状态。
5. 历史恢复 / source_lost 状态。
